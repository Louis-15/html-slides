# HTML 幻灯片内编辑功能 —— 架构优化设计方案

## 背景

将 `七选五 frontend-slides.html` 中逐步叠加的编辑功能，经过**系统性重新设计**后注入到 `html-slides` skill 的模板体系中，使生成的 HTML 文件原生自带专业级的网页内编辑能力。

本方案**不是照搬**旧代码，而是针对旧实现中暴露的以下核心问题进行架构级重写：

| 旧实现的问题 | 根因分析 |
|---|---|
| 跨会话撤销声称能用但实际无效 | 持久化数据嵌入 `<script>` 标签后，下次打开时 `JSON.parse` 的时序与 DOM 状态不同步；且 `localStorage` 与 `file:///` 协议的浏览器隔离策略冲突 |
| 撤销第一步失效（需操作两次才能亮起按钮） | `recordState()` 缺少初始快照锚点，进入编辑模式后栈内只有 0 条基准 |
| `restoreState()` 后退出编辑模式失灵 | DOM 重写后 `editableElements` NodeList 失效，`window.editor` 一度缺少全局引用 |
| 字号调节混乱 | `execCommand('fontSize')` 生成嵌套 `<span>`；调大调小操作目标不统一 |
| 代码膨胀且耦合严重 | 所有功能以过程式脚本平铺在同一个 `<script>` 块内，缺乏模块化 |

## 用户审阅要点

> [!IMPORTANT]
> **以下几个关键设计决策需要您确认后再开工：**

### 决策 1：持久化策略

旧方案声称"关闭浏览器再打开依然可以撤销"，但实际并未实现。核心原因有两个：
1. 嵌入 HTML 的 `<script type="application/json">` 虽然保存了历史栈 JSON，但浏览器通过 `file:///` 协议打开 HTML 时，JS 无法自动读取自身文件内的数据（需要用户手动 Ctrl+S 覆盖才能让数据生效）
2. `localStorage` 在 `file:///` 协议下的行为因浏览器而异（Chrome 同源共享所有 `file:///`，Edge 类似）

**新方案建议：**
- **编辑内容**（文字、格式、位置）→ 通过 `localStorage` 实时缓存，刷新不丢
- **撤销/重做栈** → 仅在**当次浏览器会话**内有效（RAM 内存），关闭即清零
- **Ctrl+S 导出** → 导出纯净 HTML，内嵌已修改的内容（不嵌入历史栈，避免文件体积膨胀）
- 如果您确实需要"关闭再打开还能撤回三天前的修改"，我会改用 IndexedDB + 文件指纹匹配方案（但复杂度会大幅增加）

> **请确认：撤销栈只在当次会话有效，是否可以接受？**

### 决策 2：编辑模式的触发体验

旧方案：左上角的热区悬停 + 按钮点击 + 键盘 E 键。  
新方案保持一致，但追加一个细节：

- 进入编辑模式后，**禁用幻灯片翻页**（鼠标滚轮、键盘上下箭头不再换页），防止编辑时误触翻页
- 仅保留工具栏上的"← 上一页 / 下一页 →"按钮进行手动翻页

> **请确认：编辑模式下禁用滚轮翻页，是否可以接受？**

### 决策 3：自定义文本框 vs 原生文本框的统一性

旧方案中，"新建的文本框"和"页面原生文本框"使用了两套不同的拖拽和删除系统（一个用 Overlay 全局覆盖层，一个用内嵌控件）。这导致：
- 交互体验不一致（原生框要先点击聚焦才出现控制按钮，自定义框悬停就有）
- 代码重复

**新方案拟统一为：**
- 所有可编辑元素在编辑模式下悬停时，统一显示右上角的 📍✖ 控件
- 拖拽和删除共用同一套事件管理器
- 区别仅在底层：原生元素使用 `transform: translate()` 偏移，自定义元素使用 `left/top` 绝对定位

> **请确认：统一控件交互方式是否可行？或者您希望原生文本框继续保持"不可拖拽、仅可删除"？**

## 各模块详细设计

### 模块 1：EditorCore（编辑模式总控）

**职责**：管理编辑模式的开/关状态，协调其他模块的激活与休眠。

| 功能 | 设计要点 |
|---|---|
| 编辑模式切换 | `toggleEditMode()` 控制 `body.classList` 和 `contenteditable` 属性 |
| 元素注册机制 | 维护 `Set<Element>` 而非 NodeList，DOM 变更后自动同步 |
| 翻页锁定 | 编辑模式下劫持滚轮/键盘事件，仅允许工具栏翻页 |
| 热区悬停 | 400ms 延迟隐藏（复用旧方案成熟逻辑） |

**与旧方案的核心区别**：
```diff
- this.editableElements = document.querySelectorAll('[data-edit-id]');
- // ↑ 静态 NodeList，DOM 重写后失效
+ this.editableSet = new Set(document.querySelectorAll('[data-edit-id]'));
+ // ↑ 可动态增删的 Set，配合 MutationObserver 自动同步

- // toggleEditMode 时直接操作旧 NodeList
+ refreshEditables() {
+     this.editableSet = new Set(document.querySelectorAll('[data-edit-id]'));
+ }
+ // toggleEditMode / restoreState 后统一调用 refreshEditables()
```

### 模块 2：RichTextToolbar（富文本工具栏）

**职责**：毛玻璃悬浮工具栏，提供所有格式化操作。

| 功能 | 优化点 |
|---|---|
| 加粗/斜体/下划线 | 继续使用 `document.execCommand`（浏览器兼容性最佳） |
| 红色删除线 | `execCommand('strikethrough')` + CSS `s { text-decoration-color: #e74c3c !important }` |
| 字号调节 | **重写**：不用 `execCommand('fontSize')`，改为直接修改容器 `style.fontSize` + 清洗内部残留的行内样式 |
| 文字颜色 / 背景高亮 | 保持调色板 UI，优化颜色数组（新增莫兰迪色系） |
| 清除格式 | `execCommand('removeFormat')` |
| Ruby 顶标 | 保持 `<ruby>/<rt>` 方案，优化间距为固定 `8px` padding-bottom（不再用 `em` 避免字号波动） |
| 工具栏布局 | `min-width: 680px` 防止换行；按钮改用纯 SVG 图标 |

**字号调节的核心重写逻辑**：
```javascript
// 旧方案：找到锚点节点的父级 data-edit-id，修改该元素的 fontSize
// 问题：如果光标在空白处或 body 上，targetEl 为 null → 无反应

// 新方案：
function changeFontSize(direction) {
    // 1. 先确定"正在编辑哪个容器"
    const container = getActiveEditContainer(); // 从 selection 或 focusedElement 推导
    if (!container) return;

    // 2. 判定是否在 <rt> 内部
    const sel = window.getSelection();
    const anchor = sel.anchorNode;
    const rtNode = (anchor?.nodeType === 3 ? anchor.parentNode : anchor)?.closest('rt');

    // 3. 精确定位目标
    const target = rtNode || container;

    // 4. 修改字号（使用 px 绝对值，避免 em 的雪崩效应）
    const current = parseFloat(getComputedStyle(target).fontSize);
    const step = 2; // 每次增减 2px
    target.style.fontSize = Math.max(10, current + direction * step) + 'px';

    // 5. 如果修改的是整个容器（非 rt），清洗内部碎片 span 的行内字号
    if (!rtNode) {
        target.querySelectorAll(':not(rt) > [style*="font-size"]').forEach(el => {
            el.style.removeProperty('font-size');
        });
    }

    // 6. 录入历史
    historyMgr.recordState();
    persistenceLayer.saveElement(container);
}
```

### 模块 3：BoxManager（元素管理器）

**职责**：统一管理所有可编辑元素的拖拽、删除、新建。

| 功能 | 设计要点 |
|---|---|
| 统一控件 UI | 悬停/聚焦任何 `[data-edit-id]` 元素时，在其右上角显示 `📍✖` 浮动控件 |
| 控件容器 | 单例 DOM 节点（`position: fixed`），动态定位到目标元素上方 |
| 拖拽引擎 | 统一使用 `PointerEvent` API（同时覆盖鼠标和触控笔），取代旧方案的 `mousedown/mousemove/mouseup` |
| 新建文本框 | `createTextBox(slideElement)` → 在当前幻灯片的 `.slide-content` / `.slide` 内添加绝对定位 div |
| 删除确认 | `confirm()` 弹窗（不变） |

**与旧方案的核心区别**：

```diff
- // 旧方案：原生元素和自定义元素有两套独立的拖拽逻辑
- // overlayCtrl (原生) + .drag-handle (自定义) × N 个 listener
+ // 新方案：一个 FloatingControls 单例，监听全局的 focusin / pointerover
+ class BoxManager {
+     constructor() {
+         this.controls = this.createControlsElement();       // 单例 fixed div
+         this.activeTarget = null;                           // 当前激活的目标元素
+         this.dragState = null;                              // { startX, startY, initPos }
+         document.body.appendChild(this.controls);
+
+         // 使用 PointerEvent + 事件委托，一次绑定覆盖所有元素
+         document.addEventListener('pointerdown', this.onPointerDown.bind(this));
+         document.addEventListener('pointermove', this.onPointerMove.bind(this));
+         document.addEventListener('pointerup', this.onPointerUp.bind(this));
+     }
+ }
```

### 模块 4：HistoryManager（撤销/重做系统）

**职责**：DOM 快照级别的全局撤销/重做。

> [!WARNING]
> **这是旧方案中 bug 最密集的模块，本次做最大幅度的重构。**

| 设计维度 | 旧方案 | 新方案 |
|---|---|---|
| 快照粒度 | 每个 `.slide` 的 `innerHTML` 组成数组 | **仅快照当前活跃幻灯片的 `innerHTML`**（减少 90% 内存消耗） |
| 初始锚点 | 遗漏 → 第一次操作无法撤销 | 进入编辑模式时**立即拍摄 Snapshot #0** |
| 防抖录制 | 500ms 全局 debounce | 300ms debounce + **显式操作（字号/颜色等）立即记录** |
| 栈容量 | 200 个微步 + 按 Session 修剪 | **每页最多 50 步**（轻量策略，避免全局 200 步 × N 页的爆炸） |
| DOM 恢复后重绑定 | 手动调用多个 rebind 函数，遗漏频出 | **统一 `rehydrate()` 入口**，一次性完成：元素注册 → 事件绑定 → 属性同步 |
| 跨会话持久化 | 声称存入 HTML 但实际无效 | **不做跨会话撤销**——撤销栈仅存 RAM；编辑内容由 PersistenceLayer 单独负责 |

**新方案的核心类**：

```javascript
class HistoryManager {
    constructor() {
        this.stacks = new Map();     // key = slideIndex, value = { undo: [], redo: [] }
        this.isRestoring = false;
        this.debounceTimer = null;
    }

    // 获取当前幻灯片索引
    get currentSlide() { return window.presentation?.currentSlideIndex ?? 0; }

    // 获取或创建某页的历史栈
    getStack(slideIndex) {
        if (!this.stacks.has(slideIndex)) {
            this.stacks.set(slideIndex, { undo: [], redo: [] });
        }
        return this.stacks.get(slideIndex);
    }

    // 拍摄当前幻灯片快照
    snapshot() {
        const slide = document.querySelectorAll('.slide')[this.currentSlide];
        return slide ? slide.innerHTML : null;
    }

    // 录入状态（isForced: 显式操作如字号变更时立即录入，跳过防抖）
    recordState(isForced = false) {
        if (this.isRestoring) return;

        if (isForced) {
            clearTimeout(this.debounceTimer);
            this._pushState();
        } else {
            clearTimeout(this.debounceTimer);
            this.debounceTimer = setTimeout(() => this._pushState(), 300);
        }
    }

    _pushState() {
        const stack = this.getStack(this.currentSlide);
        const snap = this.snapshot();
        if (!snap) return;

        // 去重：跳过与栈顶相同的快照
        if (stack.undo.length > 0 && stack.undo[stack.undo.length - 1] === snap) return;

        stack.undo.push(snap);
        stack.redo = [];  // 新操作清空重做栈

        // 容量修剪
        if (stack.undo.length > 50) stack.undo.shift();

        this.updateUI();
    }

    undo() {
        const stack = this.getStack(this.currentSlide);
        if (stack.undo.length <= 1) return;  // 至少保留基准快照

        const current = stack.undo.pop();
        stack.redo.push(current);
        this.restoreState(stack.undo[stack.undo.length - 1]);
    }

    redo() {
        const stack = this.getStack(this.currentSlide);
        if (stack.redo.length === 0) return;

        const next = stack.redo.pop();
        stack.undo.push(next);
        this.restoreState(next);
    }

    restoreState(html) {
        this.isRestoring = true;
        const slide = document.querySelectorAll('.slide')[this.currentSlide];
        if (slide) {
            slide.innerHTML = html;
            // 统一重水化入口
            window.editorCore?.refreshEditables();
            window.boxManager?.rehydrate(slide);
            window.persistenceLayer?.syncFromDOM(slide);
        }
        this.isRestoring = false;
        this.updateUI();
    }

    // 进入编辑模式时为当前页拍摄基准快照
    captureBaseline() {
        const stack = this.getStack(this.currentSlide);
        if (stack.undo.length === 0) {
            stack.undo.push(this.snapshot());
        }
    }

    updateUI() { /* 更新按钮灰度 */ }
}
```

### 模块 5：PersistenceLayer（持久化层）

**职责**：管理 `localStorage` 的读写和 `Ctrl+S` 导出。

| 功能 | 设计要点 |
|---|---|
| 内容缓存键 | `html-slides:{fileHash}:edit:{editId}` — 用文件标题哈希隔离不同课件 |
| 自定义文本框 | `html-slides:{fileHash}:custom-boxes` — JSON 数组 |
| 原生框位移/隐藏 | `html-slides:{fileHash}:native-mods` — JSON 对象 |
| 加载恢复 | 页面 `DOMContentLoaded` 时统一恢复所有缓存 |
| Ctrl+S 导出 | 克隆 `documentElement`，清洗编辑 UI → 导出纯净 HTML |
| Ctrl+S 全局化 | 无论是否在编辑模式，Ctrl+S 一律触发自定义导出（不再有条件判断） |

**与旧方案的核心区别**：

```diff
- // 旧方案：localStorage key 写死 'slide_edit_xxx'，不同课件会互相覆盖
+ // 新方案：基于文件标题生成 hash 前缀，确保隔离
+ const fileHash = hashCode(document.title);
+ const key = `html-slides:${fileHash}:edit:${editId}`;
```

**Ctrl+S 导出优化**：

```diff
- // 旧方案：在克隆节点上使用 cleanHtml.createElement（崩溃）、cleanHtml.body（undefined）
+ // 新方案：规避所有克隆节点的陷阱
+ function exportCleanHTML() {
+     const clone = document.documentElement.cloneNode(true);
+
+     // 使用 querySelector 而非 .body 访问
+     const body = clone.querySelector('body');
+
+     // 使用 document.createElement 而非 clone.createElement
+     // ... 后续清洗
+
+     const html = '<!DOCTYPE html>\n' + clone.outerHTML;
+     downloadBlob(html, 'text/html', document.title + '.html');
+ }
```
