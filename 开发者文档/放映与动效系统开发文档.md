## 一、系统定位与设计初衷

在 HTML 幻灯片生态中，如果说 Zone2 布局管的是“空间与排版”，Editor 引擎管的是“破坏与重建”，那么放映系统（Runtime）掌管的就是**时间维度**与**视觉动效**。

它由 JavaScript 引擎（`slides-runtime.js`）与全局 CSS 关键帧动画（`components.css` 中的 `@keyframes` 及各主题包中的绑定）共同构成。

**这一层的核心职责是绝对纯粹的：**
只负责控制哪一张幻灯片（Slide）该出现在观众面前、哪一个碎片（Step）该按顺序点亮，以及负责协调极其脆弱的入场动画体系，确保放映过程丝滑、不串台。

## 二、架构总览与时间流逝机制

整个放映动效系统的运转，依托于一个极其极简的**“类开关协议”**：

```text
┌──────────────────────────────────────────────────────────────┐
│                    slides-runtime.js                         │
│  [键盘监听] / [滚轮防抖] / [小圆点点击]                        │
└──────────────────────┬───────────────────────────────────────┘
                       │ 唯一通信信使：.active 类
┌──────────────────────┴───────────────────────────────────────┐
│                    DOM 层级状态切换                           │
│  <div class="slide active">   → 当前放映页 (Z-index 最高)      │
│  <div class="slide">          → 背景沉睡页 (Z-index 隐藏)      │
└──────────────────────┬───────────────────────────────────────┘
                       │ 级联触发 CSS 动画
┌──────────────────────┴───────────────────────────────────────┐
│                    CSS 渲染管线 (主题 / components.css)      │
│  .slide.active .anim-1 { animation: slideUp ... }            │
│  .slide.active .step.active { animation: fadeIn ... }        │
└──────────────────────────────────────────────────────────────┘
```

**设计铁律：JS 绝不直接操作元素的几何属性进行翻页/入场计算。** 所有的动画编排、时间轴延迟（Delay），全部交给浏览器的 CSS 渲染管线处理。JS 只充当拨动开关的“手”。

### 2.1 翻页底座机制：Z轴透明叠片架构
与传统的强制滚动 (`scrollIntoView`) 不同，我们的引擎彻底抛弃了滚动条物理位移，而是采用了**“Z轴叠片（Cross-fade Stack）”**架构：
1. **全员绝对定位层叠**：在 `components.css` 中，所有的 `.slide` 都被赋予了 `position: absolute; inset: 0;`。这意味着如果一份课件有 50 页，它们在 DOM 物理结构下是完全叠在同一个 X/Y 坐标上的。
2. **默认透明休眠**：所有的基础 `.slide` 均自带 `opacity: 0; pointer-events: none;`。这就如同 50 张全透明的玻璃板。没有焦点，没有阻挡，没有内容渲染压力。
3. **点亮即现身**：当且仅当 JS 的引擎为幻灯片加封 `.active` 王冠时，CSS 选择器 `.slide.active` 赋予其 `opacity: 1; pointer-events: auto;`。此时，带有 `transition: opacity 0.45s` 缓冲属性的页面将以极其丝滑的交叉渐变（Cross-fade）形式跃然荧幕，并一并唤醒该玻璃板内所有随属组件的入场小动画。

## 三、三维导航与步进系统 (Stepping)

幻灯片的放映不仅仅是 2D 的页与页切换，内部还嵌套了步骤进阶（Fragment / Stepping）。

### 3.1 导航防抖与降噪
原生的 `wheel`（滚轮）和 `touchmove`（触摸）事件触发频率高得惊人。为了防范用户稍微滑一下鼠标滚轮连翻十页的灾难：
- 引擎内置了严格的**时间戳防抖锁（Debounce Lock）**，两次翻页之间存在物理时间冷却（例如 500ms）。
- 绝不因动画没播完而卡死逻辑，翻页指令是最高优先级的，随时可以直接掐断当前动画强行跳转新界面。

### 3.2 矩阵步进算法 (.step)
当当前 `.slide` 拥有子节点带有 `.step` 类时，放映逻辑将从二维（翻页）降维至一维（步进）：
1. 默认情况下，没有挂载 `.active` 的 `.step` 元素在 CSS 层表现为 `opacity: 0` 和 `pointer-events: none`（存在占位，但视觉不可见且阻断交互）。
2. 每按一次“下一页”快捷键，如果当前页存在未显示的 `.step`，JS 会拦截翻页请求，转为寻找下一个 `.step` 并为其追加 `.active`。
3. 追加的瞬间，CSS 选择器 `.step.active` 被激活，触发单独绑定在其身上的入场小动画。

## 四、入场动效系统与 CSS 特异性架构

为了让数以百计的组件不用被挨个手动调配动画，我们采取了**阶梯式联级动画宣告**。

### 4.1 动画库 (components.css)
定义了基础物理学规律库，比如弹性升起、玻璃破裂、极速滑入等纯关键帧定义（`@keyframes slideUpElastic`）。

### 4.2 主题绑定层 (themes/*.css)
主题掌控了哪些组件该在什么时候入场。
通过设定高度的 CSS 特异性：
```css
.slide.active .layout-single > * {
    animation: fadeUp 0.8s ease forwards;
}
.slide.active .anim-1 { animation-delay: 0.1s; }
.slide.active .anim-2 { animation-delay: 0.2s; }
```
**原理**：只要父元素 `.slide` 失去了 `.active`，由于选择器条件不成立，子元素身上的 animation 属性瞬间瓦解。而在下一次被重新赋予 `.active` 时，由于规则重新生效，浏览器将其视作一个处于初始状态的全新节点，再次从 0 毫秒开始放映一遍入场动画。

## 五、核心避坑实录：动画与编辑的“不死之争”

在打通 `editor-core` (动态编译引擎) 与此放映引擎时，最剧烈的冲突爆发了。

### 🔥 致命冲突：DOM 重挂载导致的疯狂重播
**背景描述**：
当初次进入编辑模式时，编辑器需要在底层为每个支持编辑的组件套上一个 `.native-edit-wrap` 隔绝层框。在浏览器视角，这属于把旧节点拔出来重新塞回去（DOM Re-mount）。紧接着退出编辑模式时，又需要把框子拆除再塞回去。
**灾难现象**：
每一次进出编辑模式，所有的幻灯片由于被视为“新挂载”，无视了之前已经播放完毕的状态，集体重新重播了一遍轰轰烈烈的入场动画，导致排版发生极严重的视觉震荡（闪烁位移）。

### 🛡️ 高级抑制协议的诞生
为了镇压原生底层的这股反扑，我们放弃了暴力的 `animation: none`，创立了**“运行时动画降维冻结协议”**。技术路径如下：

#### 策略 1：Z轴阻绝机制 (CSS Paused)
```css
/* 压制规则优先级超越普通主题动画 (0,3,1) vs (0,3,0) */
html.editor-mode .slide.active *,
html.editor-mode .slide.active *::before,
html.editor-mode .slide.active *::after {
    animation-play-state: paused !important;
}
```
不再采用 `none` 把动画抹除（那会导致退出时动画视为新生而重播），而是采用**暂停策略** (paused)。
这使得所有在编辑环境下的元素，其动画状态定格在了当前的最终结果上。它就在那里，但被按了暂定键。

#### 策略 2：瞬步击杀机制 (Web Animations API)
然而，首次注入 Wrapper 以及进行 Undo/Redo (撤销重做引发 `innerHTML` 重写) 时的瞬间，新的 DOM 是没有历史积累时间轴的，一旦建立它们会在毫秒内尝试从头播放。
借此引入了现代 JS Web 动效抹杀能力：
```javascript
if (slide.getAnimations) {
    try {
        slide.getAnimations({ subtree: true }).forEach(a => a.finish());
    } catch(e) {}
}
```
这句话能一键抓取发生重排后试图启动的所有子帧动画，并在第 1 毫秒时利用 `.finish()` 将它们的时间轴直接拉拽到大结局尽头。

这两套连招让动态组件引擎与静态动效引擎签下了互补协议，让课件实现了“编辑动作瞬间完成，毫无花哨残影”的超工业级所见即所得体验。
