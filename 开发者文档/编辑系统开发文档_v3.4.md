## 四、核心技术原理详解

### 4.1 装饰格式操作的完整流水线

这是整个编辑系统中最复杂的部分。浏览器的 `Range.extractContents()` 有一个关键行为：**当选区完全包含在一个 span 内部时，它只提取文本节点，不带走包裹的 span**。这导致了所有装饰（删除线/下划线）操作的不稳定。

#### 4.1.1 问题场景示例

```
DOM: <p><span style="text-decoration:line-through">ABCDEFGH</span></p>
用户选中 "CDE"，想只去掉 CDE 的删除线
```

**错误流程**（v3.0 之前）：
```
1. extractContents() → 只抽出文本 "CDE"，span 还在（变成 <span>ABFGH</span>）
2. _cleanDecoration(fragment) → fragment 里没有 span → 清不掉
3. range.insertNode(fragment) → "CDE" 被插回原 span 内部 → 无效果
```

#### 4.1.2 修复方案：三阶段流水线

```
阶段 1: _isRangeFullyDecorated() — 预检活 DOM
阶段 2: _splitDecoSpanAtRange()  — 在选区边界劈开装饰 span
阶段 3: extract + clean + insert — 标准操作
```

**完整流程**：

```javascript
_toggleDecoration(decorationType, decoColor) {
    // 1. 预检：在活 DOM 上判断选区是否全部已有装饰
    var allDecorated = this._isRangeFullyDecorated(range, container, decorationType);
    
    // 2. 劈开：如果选区在装饰 span 内部，在边界处劈开
    this._splitDecoSpanAtRange(range, container, decorationType);
    // 劈开后 DOM 变为: <span>AB</span><span>CDE</span><span>FGH</span>
    //                   (有装饰)     (无装饰-锚点) (有装饰)
    
    // 3. 标准操作
    var fragment = range.extractContents();  // 现在能正常抽出 "CDE"
    this._cleanDecoration(fragment);         // 清理 fragment 内残留
    this._cleanEmptyDecoSpans(container);    // 清理空壳 span
    
    if (allDecorated) {
        range.insertNode(fragment);   // 取消：裸插回
    } else {
        // 添加：包裹新装饰
        var span = document.createElement('span');
        span.style.textDecoration = decorationType;
        span.appendChild(fragment);
        range.insertNode(span);
    }
    container.normalize();  // 合并碎片文本节点
}
```

#### 4.1.3 `_splitDecoSpanAtRange` 的劈裂算法

```
输入: <span style="line-through">ABCDEFGH</span>，选区 = "CDE"

步骤 1: 提取「选区之后」的内容
  afterRange = [E后..span末尾] → extractContents() → "FGH"
  span 变为: <span>ABCDE</span>

步骤 2: 提取「选中」的内容
  range = [C..E] → extractContents() → "CDE"  
  span 变为: <span>AB</span>

步骤 3: 插入「之后」部分（保留装饰）
  克隆 span 的样式 → <span style="line-through">FGH</span>

步骤 4: 插入「选中」部分（保留无样式 span 做选区锚点）
  <span>CDE</span>  ← 无 style，不影响渲染但能稳定锚定选区

步骤 5: 设置选区指向锚点 span 的内容

结果: <span style="line-through">AB</span>
      <span>CDE</span>          ← 选区在此
      <span style="line-through">FGH</span>
```

> [!IMPORTANT]
> **不能解包锚点 span**！之前尝试过解包（`while(span.firstChild) parent.insertBefore(...)`），但浏览器会在此时**丢弃选区**，导致后续所有操作失效。保留无样式 span 不影响视觉渲染，且能稳定锚定选区。`container.normalize()` 会在最终操作后合并碎片。

#### 4.1.4 `_isRangeFullyDecorated` 的检测逻辑

```javascript
// 快速路径: 如果选区的公共祖先就在装饰 span 内 → 全部已装饰
var checkEl = anchorElement;
while (checkEl !== container) {
    if (checkEl.style.textDecoration.includes(decorationType)) return true;
    checkEl = checkEl.parentNode;
}

// 完整路径: 遍历选区内每个文本节点，检查其祖先链
// 全部有 → true（取消模式）
// 任一没有 → false（应用模式）
```

#### 4.1.5 切换逻辑总结

| 选区状态 | 操作 | 结果 |
|----------|------|------|
| 全部无装饰 | 点击按钮 | 清旧 + 整体包裹新装饰 |
| 全部有装饰 | 点击按钮 | 劈开 + 取消装饰 |
| 部分有装饰 | 点击按钮 | 清旧 + 整体包裹新装饰（一步到位） |
| 选区在装饰内部（部分取消） | 点击按钮 | 劈开 → 取消选中部分 → 其余保留 |

### 4.2 字号调节的选区管理

字号调节使用 span 包裹方式（非 `execCommand('fontSize')`），因为 execCommand 会生成不可控的嵌套 `<font>` 标签。

```javascript
changeFontSize(dir) {
    // 有选区 → 包裹 span
    var span = document.createElement('span');
    span.style.fontSize = (currentFS + dir * 2) + 'px';
    span.style.lineHeight = 'inherit';  // 防止行高变化导致布局偏移
    span.appendChild(range.extractContents());
    range.insertNode(span);
    
    // 关键：重建选区指向新 span，否则连续点击 A+ 时选区丢失
    var newRange = document.createRange();
    newRange.selectNodeContents(span);
    sel.removeAllRanges();
    sel.addRange(newRange);
    this.savedRange = newRange;
    
    // 无选区 → 改整个容器 + 清洗子元素碎片字号
}
```

### 4.3 选区保存与恢复

所有工具栏按钮使用 `pointerdown` 事件（而非 `click`），因为 `click` 会导致 `contenteditable` 元素失焦从而丢失选区。

```javascript
// 全局监听 selectionchange，保存选区到 savedRange
document.addEventListener('selectionchange', function() {
    var range = sel.getRangeAt(0);
    var el = range.commonAncestorContainer;
    // 仅保存在 contenteditable 内的选区
    if (el.closest('[contenteditable="true"]')) {
        self.savedRange = range;
    }
});

// 工具栏按钮点击时先恢复选区
button.addEventListener('pointerdown', function(e) {
    e.preventDefault();  // 阻止 blur
    RichTextToolbar.restoreSelection();  // 恢复 savedRange
    RichTextToolbar.execAndRecord(cmd);  // 执行命令
});
```

| 数据类型 | 存储方式 | Key 命名规则 |

| 文本内容 | localStorage | `hslides:{fileHash}:e:{editId}` |
| 自定义文本框 | localStorage | `hslides:{fileHash}:boxes` |

### 4.5 编辑模式的翻页锁定

```javascript
// 在编辑模式下，使用 capture 阶段拦截所有翻页键
document.addEventListener('keydown', function(e) {
    if (editorCore._navLocked) {
        var navKeys = ['ArrowDown','ArrowUp',' ','PageDown','PageUp','Home','End'];
        if (navKeys.includes(e.key)) {
            e.stopPropagation();
            // 如果在 contenteditable 内，允许方向键移动光标
            if (!e.target.isContentEditable) e.preventDefault();
        }
    }
}, true);  // ← capture:true 在冒泡之前拦截

// 滚轮同理
document.addEventListener('wheel', function(e) {
    if (editorCore._navLocked) e.stopPropagation();
}, true);
```

## 六、开发注意事项

### 6.1 savedRange 的生命周期

`savedRange` 是整个格式化系统的命脉。以下操作会使其失效：
- `range.extractContents()` — 提取后原 range 坍缩
- `range.insertNode()` — 插入后 range 指向变化
- DOM 结构变更 — 节点移动/删除后 range 引用失效

**规则**：每次 DOM 操作后，必须用 `document.createRange()` + `selectNodeContents()` 重建选区，并更新 `this.savedRange`。

### 6.2 extractContents 的陷阱

`Range.extractContents()` 的行为取决于选区是否跨越了元素边界：
- **跨边界**：正确提取包含标签的 DocumentFragment
- **不跨边界**（选区完全在一个元素内）：只提取文本节点，包裹标签留在原位

这是 v3.0 → v3.4 所有 bug 的根因。**解决方案**：在 `extractContents` 之前调用 `_splitDecoSpanAtRange()` 先劈开包裹元素。

### 6.3 container.normalize() 的作用

每次格式操作后调用 `container.normalize()`，它会：
- 合并相邻的文本节点
- 移除空文本节点

这能消除反复操作产生的 DOM 碎片，保持 HTML 结构干净。

## 七、设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 撤销栈范围 | 仅当次会话 | localStorage 对 file:/// 协议不稳定，IndexedDB 复杂度过高 |
| 编辑模式翻页 | 全部锁定 | 空格键翻页在编辑时极其干扰（打空格变翻页） |
| 文本框控件 | 统一 📍✖ | 消除原生框和自定义框的体验差异 |
| 拖拽 API | PointerEvent | 同时覆盖鼠标/触控笔/触屏 |
| 删除线实现 | 自定义 span | execCommand 不支持自定义 text-decoration-color |
| 下划线实现 | 自定义 span | 需要支持多种颜色 |
| 字号调节 | 自定义 span | execCommand('fontSize') 生成不可控的嵌套标签 |
| 装饰格式解包 | 不解包锚点 span | 解包导致浏览器丢失选区 |
| 加粗/斜体 | execCommand | 浏览器原生处理最稳定，自带 toggle 和部分选区分割能力 |
| 文字颜色/高亮 | execCommand | 同上，原生处理 span 分割逻辑完善 |
