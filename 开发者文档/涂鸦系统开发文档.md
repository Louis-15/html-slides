## 一、系统定位与设计初衷

在 HTML 幻灯片（html-slides）的演示与互动场景中，教师与观众需要突破极其克制的标准 DOM 文本块拖拽，获得一层**绝对自由的手写涂鸦白板**。

如果尝试将每一笔鼠标/手写笔拖曳记录为带有内联 CSS 和绝对定位的 SVG 节点插入到现有的 `contenteditable` 层，将会引发灾难：
- 极高的重排（Reflow）性能损耗，导致文字选区拖拽极其卡顿。
- DOM 结构由于混入了百万节点的坐标垃圾，无法实现纯净 HTML 导出及持久化（PersistenceLayer 无从下手）。
- 原生指针事件会产生重叠碰撞。

**核心解决方案：物理隔离的双轴系统。**
涂鸦层 (`DoodleRuntime`) 被设计为覆盖在文本 DOM 层上方的一块“绝对纯净的高透光靶标板” (`<canvas>`)。

---

## 二、系统架构与运行机制

涂鸦模块由核心类与底层 Canvas 渲染管线构成，它是作为一个**独立模块插槽**载入编辑系统的：

```text
┌────────────────────────────────────────────────────────┐
│                   DoodleRuntime (涂鸦引擎核心)            │
│  isDrawingMode 显隐控制  │  PointerEvent 拦截  │ 路径算法   │
└──────────────────────┬─────────────────────────────────┘
                       │ 接管指针落点
┌──────────────────────┴─────────────────────────────────┐
│                 Overlay Canvas (覆盖层画板)               │
│  <canvas> (width: 100vw, height: 100vh)                │
│  CSS: pointer-events: none (平时) -> auto (激活时)        │
└────────────────────────────────────────────────────────┘
```

### 2.1 穿透与拦截机制 (Pointer Events Check)

这是一个极其优雅的隔离防波堤。
在日常编辑/播放幻灯片时，Canvas 层依然绝对存在，但被赋予 `pointer-events: none`，所有的鼠标点击、文字拖拽都会穿透 Canvas 正常与底层 `BoxManager` 及 `RichTextToolbar` 发生反应。

一旦用户通过左下角的按钮激活**涂鸦模式 (Doodle Mode)**：
1. `DoodleRuntime` 会接管 `<canvas>`，将其 `pointer-events` 全盘变更为 `auto` / `all`。
2. 此时 `<canvas>` 成为最高层级的玻璃遮罩，拦截所有点击与滑动物理事件。
3. 底层的 DOM 完全不会受到鼠标影响，彻底切断了富文本层面的误触与混乱。

## 三、核心技术实现细节

### 3.1 路径平滑算法 
为了避免在电子白板上“字迹发虚带锯齿”或出现锐利的拐角，Doodle 必须使用二次/三次**贝塞尔曲线**拟合用户的拖动指针点，而不能简单地使用原始事件的 X/Y 点集用 `lineTo` 画多边形。

### 3.2 压感模拟
在收到触控板或 Apple Pencil 等电磁手写笔传代的 `PointerEvent` 时，引擎实时读取 `e.pressure`。如果设备支持，线条的绘制粗细将会以动态画刷的形式，实现粗细自然的“笔锋”过渡。

### 3.3 橡皮擦 (Eraser) 的透明度合成
由于在 Canvas 上实现“局部橡皮”不能只是简单地画一层白胶布（因为画布底下还得看到本来的幻灯片文本），因此系统调用 `globalCompositeOperation = 'destination-out'` 达到类似物理擦除橡皮的效果，把已涂画的部分真真切切地变为透明抠空。

## 四、外挂式的涂鸦导入导出体系 (.doodle)

经过多轮架构研判与沙箱限制考量，为了让幻灯片源文件维持极致的轻盈纯净，本系统最终**摒弃了伴随 HTML 发行包混合打包涂鸦的传统方案**，转而采用一种更优雅的“旁路数据解耦 (Sidecar)”机制。

### 4.1 独立记忆文件导出导出
现在，在涂鸦专属控制面板中接入了涂鸦脱机缓存通道：
- **物理介质与格式**：剥离于 DOM 之外，涂鸦数据被单独萃取打包成具有实体专有后缀名的 `*.doodle` 文件 (其实体本质为记录 SVG 座标地图的 JSON)。
- **File System Access API 深度应用**：系统调用了新一代浏览器的文件沙箱逃脱 API (`window.showSaveFilePicker`)。用户点击导出时，它不再将乱码默默堆积在下载库，而是直接唤醒操作系统原生的“另存为”资源管理器窗口，允许使用者随心指定存放路径甚至就是将其保存在当前所演示 HTML 的同级目录。
- **智能伴生预命名**：导出时，引擎具有强嗅探性，会自动剥离并截取当前执行的 HTML 的真实主体名称（并自动剪去了`.html`），例如自动帮你填入默认下载名为 `[原课件文件名]_涂鸦板书.doodle`。
- **静默沉降**：这种导出的触发是一种完全的备份过程，在成功保存数据后并不会对当前的视口缓存发起暴力打扫或强制刷新，屏幕笔痕依然完好无损伴随你的演示。

### 4.2 极客范的数据热加载机制
作为闭环，系统为导出的这个 .doodle 数据晶体文件注入了两种兼顾传统与未来演示习惯的恢复机制（且全程无需刷新页面）：
1. **显式恢复通道**：通过点击下沉极简设计的独立导入按钮，可随时随地触发文件拾取弹窗注入黑板存档。
2. **全局焦点拖拽 (Drag & Drop) 感知**：极大提升演讲时的装配体验。无论你在什么模块下，只需要通过鼠标手势划将任何一枚具备 `*.doodle` 后缀的内容包强行甩到浏览器的任何视区，系统都会在触碰松手的瞬间将所有记录即刻平铺上墙，再现现场推演。

一句话总结设计哲学：**HTML 专任无尘分发散播介质；而 `*.doodle` ，专作重装推演记忆的贴身副盘。**
