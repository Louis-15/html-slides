## 一、系统定位与设计初衷

在 HTML 幻灯片的规模化量产过程中，AI 需要能够通过一套**声明式的布局语法**，将教学内容按需注入不同空间分区（单列、双栏、三栏、四宫格等），同时又需要一组**零耦合的纯组件**可以被自由地嵌入任何分区插槽。

如果任由布局与组件混作一团——就像旧版那样，每种组件自带一套完整的页面结构、固定的 padding 与 grid 定义——后果是灾难性的：

- **修改间距就炸全场**：你想把某个组件从单列挪到双栏，发现它身上硬编码了一套 `grid-template-columns`，和双栏布局的 grid 发生了物理冲突，布局直接塌方。
- **组件膨胀到不可维护**：每个组件都得自己管排版，一个翻转卡片就携带了 `.flip-grid`、`.flip-bounce-wrap` 等三层嵌套包装器，想换个排列方式必须改组件源码。
- **主题切换形同虚设**：颜色硬写在组件里（`.stat-number.blue { color: #58a6ff; }`），换了主题文件也只是换了个背景色，所有组件上的强调色依然是旧主题的蓝紫。

**核心解决方案：布局与组件的物理解耦。**

Zone 2 被重构为一个双层架构：上层是纯空间分区的**布局系统**（8 种模式 + 1 变体），下层是不含任何布局职责的 **12 个纯组件**。两层之间通过一个极简的容器约定衔接——组件 `width: 100%` 撑满插槽，`margin: 0` 由布局 `gap` 控制间距。

│  └────────────────────────┬────────────────────────────────┘  │
│                           │ 颜色注入 (CSS 变量)                │
│  ┌────────────────────────┴────────────────────────────────┐  │
│  │ 主题层 (themes/*.css)                                     │  │
│  │  :root { --brand-primary; --accent-blue; ... }            │  │

│  │  职责：颜色、字体、品牌标识。绝不涉及组件结构。              │  │
│  └─────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────┘

### 2.1 三层职责的铁律

| 层级 | 文件 | 管什么 | 绝不碰什么 |
|------|------|--------|-----------|
| **布局层** | `zone2-content.css`（前半） | `grid-template-columns`、`gap`、`padding`、`align-items` | 颜色、字号、圆角、阴影 |
| **组件层** | `zone2-content.css`（后半） | `border-radius`、`font-size`、`transition`、`overflow` | `grid-template-*`、`margin`（统一为 0） |
| **主题层** | `themes/xindongfang-green.css` 等 | CSS 变量（`--brand-primary`）、字体导入 | 选择器、`!important`、HTML 结构 |

### 2.2 连接约定

布局与组件之间只靠一个极简规则连接：

```css
/* 组件端 — 交出全部空间控制权 */
.card, .stat-card, .highlight-card, ... {
  width: 100%;        /* 撑满插槽 */
  box-sizing: border-box;
  max-width: min(100%, xxxpx);  /* 双重限制防溢出 */
  /* margin: 0 — 间距全部由布局 gap 统管 */
}

/* 布局端 — 只管空间分区 */
.slide-content.layout-2col {
  display: grid;
  grid-template-columns: 1fr 1fr;
  gap: 32px;           /* 唯一的间距来源 */
}
```

这一约定的核心好处：**任何组件可以被放进任何布局插槽里，不需要改一行代码。**

## 三、布局系统 (8 + 1 变体)

### 3.1 布局清单

| 编号 | 类名 | 实现方式 | 说明 |
|------|------|---------|------|
| 0 | `layout-title` | flexbox 垂直居中 | 封面页 |
| 1 | `layout-single` | flexbox 垂直 | 单列流式排列 |
| 2 | `layout-2col` | grid 1fr 1fr | 等宽双栏 |
| 2v | `layout-2col.compare` | grid + `::after` 伪元素 | 双栏对比（自动插入 VS 圆圈） |
| 3 | `layout-2col-wide-left` | grid 1.85fr 1fr | 左宽右窄 |
| 4 | `layout-2col-wide-right` | grid 1fr 1.85fr | 左窄右宽 |
| 5 | `layout-3col` | grid repeat(3, 1fr) | 三栏等宽 |
| 6 | `layout-2row` | flexbox 垂直 | 上下双行 |
| 7 | `layout-grid-2x2` | grid 1fr 1fr（不设 rows） | 四宫格，行高由内容决定 |

### 3.2 关键设计决策：四宫格的动态行高

旧版四宫格使用 `grid-template-rows: 1fr 1fr` 将两行等高。但当一个 cell 里的折叠卡片被展开时，等高会导致卡片被强行截断或是大量空白。

**解决方案**：移除 `grid-template-rows`，让 CSS Grid 根据内容自动计算每行的高度：

```css
.slide-content.layout-grid-2x2 {
  grid-template-columns: 1fr 1fr;
  /* 注意：故意不设 grid-template-rows，让行高由内容动态决定 */
  gap: 24px;
}
```

### 3.3 对比变体的 ::after 伪元素

`layout-2col.compare` 不是一个独立的布局模式，而是在标准双栏上追加 `.compare` 类。这层追加利用 `::after` 伪元素在两栏正中间画一个带脉冲光晕动画的 VS 圆圈，不占用任何 DOM 位置，不干扰布局。

### 4.2 翻转卡片的结构简化史

旧版翻转卡片是整个重构中砍得最狠的一个组件。原来要写出 4 个翻转卡片，你的 DOM 树是这样的：

```text
旧版：
.flip-grid → .flip-bounce-wrap.bounce-1 → .flip-card → (.flip-front + .flip-back)
.flip-grid → .flip-bounce-wrap.bounce-2 → .flip-card → (.flip-front + .flip-back)
...

两层无意义的包装器 (.flip-grid + .flip-bounce-wrap)！
```

**为什么砍掉？**
- `.flip-grid` 本质是在组件内部硬编码了一个 2×2 grid 布局——这违反了"组件绝不涉及布局"的铁律。如果你需要 3 个卡片排一行，就得写一个新的 `.flip-grid-3`，荒谬。
- `.flip-bounce-wrap` 是动画载体。但动画类 `bounce-1` 完全可以直接打在 `.cell` 插槽容器上（由 `components.css` 定义），不需要额外的包装器。
- 按钮类从 `.card-action-btn` 拆分为 `.flip-action-btn` 和 `.collapse-action-btn`——因为翻转按钮和折叠按钮的视觉行为完全不同（翻转是循环箭头，折叠是向下箭头需要旋转 180°）。

**新版结构**：每个 `.flip-card` 独立存在，直接嵌入布局插槽。

```text
新版：
.slide-content.layout-grid-2x2
  └─ .cell → .flip-card → (.flip-front + .flip-back)
  └─ .cell → .flip-card → (.flip-front + .flip-back)
  └─ .cell → .flip-card → (.flip-front + .flip-back)
  └─ .cell → .flip-card → (.flip-front + .flip-back)
```

## 五、上下文色彩嗅探系统 (Contextual Gradient Theming)

这是本次重构中最精巧的工程设计。它解决了一个痛点：**同一个组件在不同栏位中应该自动呈现不同的品牌色，而不是让 AI 逐个手动指定颜色类。**

### 5.1 设计思路

核心思想是利用 **CSS 变量作用域**——一个 CSS 变量的值由其最近祖先元素上的声明决定。

```text
主题文件中声明:
.col:nth-child(3n+1) { --context-gradient: 绿→蓝; }
.col:nth-child(3n+2) { --context-gradient: 黄→橙; }
.col:nth-child(3n)   { --context-gradient: 蓝→紫; }

组件中消费:
.stat-number { background: var(--context-gradient, 默认渐变); }
.highlight-title { background: var(--context-gradient, 默认渐变); }
```

**结果**：当三个 `.stat-card` 分别放在 `layout-3col` 的三个 `.col` 中时，它们的数字自动呈现出不同的渐变色——第一个绿蓝、第二个黄橙、第三个蓝紫——**不需要在 HTML 中加任何颜色类**。

### 5.2 变量回退链

`var(--context-gradient, linear-gradient(90deg, var(--brand-primary, var(--accent-blue)), var(--accent-blue, var(--accent-purple))))`

这是一条三层嵌套的回退链：
1. 如果当前元素在多栏布局中 → 使用主题文件注入的 `--context-gradient` ✅
2. 如果在单栏中（没有 `.col` 祖先，`--context-gradient` 未定义）→ 使用 `linear-gradient(var(--brand-primary), var(--accent-blue))` ✅
3. 如果主题文件也没有定义 `--brand-primary` → 最终回退到 `--accent-blue` 和 `--accent-purple` ✅

### 5.3 手动覆盖后门

如果 AI 需要强制指定颜色（不服从上下文嗅探），可以在 `.stat-number` 上追加修饰类：

```html
<!-- 强制使用橙色渐变 -->
<div class="stat-number orange">42%</div>
```

这些修饰类（`.blue`、`.green`、`.orange`、`.purple`）直接用 `background: linear-gradient(...)` 覆盖了 `var(--context-gradient)` 的声明，优先级更高。

## 六、开发中踩过的坑

### 🕳️ 坑 1：CSS 组合选择器多余空格导致嗅探失效

**症状**：主题文件中写了 `nth-child` 规则，但组件的数字颜色始终走默认色，嗅探不生效。

**原因**：选择器写成了 `.slide-content[class*="layout-"] > .col :nth-child(3n+1)`——注意 `.col` 后面多了一个空格，变成了"`.col` 的后代中第 N 个子元素"，而不是"`.col` 本身是第 N 个子元素"。

**修复**：确保选择器紧凑无空格：`.slide-content[class*="layout-"] > .col:nth-child(3n+1)`。

**避坑规则**：凡是涉及 `nth-child` 的选择器，写完后务必用浏览器 DevTools 检查元素是否真的匹配上了。

### 🕳️ 坑 2：`-webkit-background-clip: text` 叠加 `color` 冲突

**症状**：渐变文字在某些组件上不显示，文字变成透明不可见。

**原因**：旧的修饰类（`.blue`）设置了 `color: #58a6ff`（纯色），而新的渐变渲染需要 `-webkit-text-fill-color: transparent` + `background-clip: text`。两者冲突时 `color` 会覆盖 `text-fill-color`（在部分浏览器中）。

**修复**：所有修饰类追加 `color: unset` 来清除旧的 `color` 遗产。

**避坑规则**：一旦使用 `background-clip: text` 渐变文字方案，就必须同时声明 `color: unset` 来确保 `text-fill-color: transparent` 生效。

### 🕳️ 坑 3：旧包装器残留导致 hover 事件穿透

**症状**：重构后的翻转卡片在 `layout-grid-2x2` 中点击翻转偶尔无反应。

**原因**：HTML 中残留了旧的 `.flip-bounce-wrap` 包装器，导致点击事件被包装器拦截而没有传递到 `.flip-card`。

**修复**：彻底删除 `.flip-bounce-wrap` 包装器，将 `onclick` 绑定在 `.flip-action-btn` 按钮上（而非 `.flip-card` 自身），通过 `this.closest('.flip-card').classList.toggle('flipped')` 向上冒泡查找。

**避坑规则**：点击事件的触发源应该始终是最内层的交互元素（按钮），用 `closest()` 向上查找目标。不要把 `onclick` 放在组件根元素上。

### 🕳️ 坑 4：脚本编码错乱摧毁整个预览页

**症状**：生成脚本运行成功，但生成的 HTML 文件在浏览器中一片空白。

**原因**：PowerShell 的 `Set-Content` 在读写包含中文的 JS 文件时，默认使用系统编码（GBK）而非 UTF-8，导致所有中文字符变成乱码。乱码字符碰巧碰坏了 JS 的字符串引号闭合，整个脚本变成了语法错误。即使 Node.js 勉强执行了（因为报错位置在数据区之后），输出的 HTML 中所有 `</div>` 闭合标签也被乱码吃掉了，浏览器无法解析。

**修复**：始终使用 `-Encoding UTF8` 参数：`Get-Content -Encoding UTF8` / `Set-Content -Encoding UTF8`。

**避坑规则**：在 Windows + PowerShell 环境下操作任何包含非 ASCII 字符的文件时，**每一次**读写都必须显式指定 `-Encoding UTF8`。不要信任默认编码。

## 八、后续演进方向

1. **响应式断点**：当前布局在窄屏上不做自动坍缩（因为课件主要在 16:9 屏幕显示）。如果未来需要手机端查看，需要为每个 `layout-*` 添加 `@media` 查询将多栏降级为单栏。
2. **Zone 扩展**：如果未来出现第四个 Zone（如互动答题区），应按相同的物理隔离原则创建 `zone4-interaction.css`，绝不与 Zone 2 混写。
3. **更多主题**：上下文色彩嗅探系统是主题无关的——只要新主题文件也定义了 `.col:nth-child()` 上的 `--context-gradient`，所有组件就会自动适配新的品牌色。

一句话总结设计哲学：**布局只管空间，组件只管自己，颜色只听主题——三层各自为政，互不侵犯。**
